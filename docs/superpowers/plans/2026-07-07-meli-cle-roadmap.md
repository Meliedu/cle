# Meli × CLE Checkpoint Loop — Master Roadmap (Cross-Session Tracker)

> **For agentic workers:** This is the CROSS-SESSION CONTRACT, not a task-level plan. Each phase gets its own detailed plan file (`2026-MM-DD-meli-cle-pN-*.md`) written with the `superpowers:writing-plans` skill, then executed with `superpowers:subagent-driven-development` (recommended) or `superpowers:executing-plans`. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Build the checkpoint-centered course loop (spec: `docs/superpowers/specs/2026-07-06-meli-cle-checkpoint-loop-design.md`) end-to-end for teacher + student, across 8 independently-shippable phases run in separate Claude sessions.

**Architecture:** Extend-in-place on the existing FastAPI + Next.js 16 monorepo. New domain layer (checkpoints, attendance/QR, activities, score policy, work-item checklist spine, readiness, reports, course-memory API) rides the existing evidence seam (`learning_events` → notes → review → follow-ups → mastery). Role-scoped frontend IA (`/teacher`, `/student`) replaces the shared `/dashboard`. CLE specifics live in a pilot config module.

**Tech Stack:** FastAPI, SQLAlchemy 2.0 async, Alembic, Postgres 17 + pgvector, DB task queue; Next.js 16 App Router, React 19, TanStack Query, Better Auth (JWT + genericOAuth-ready), next-intl, Playwright, pytest.

NOTE: the Windows full-suite pytest run suffers an event-loop cascade under load (asyncpg + selector loop) that inflates the raw failure count — per-file/sharded runs are clean; only the 4 known pre-existing files (test_alerts_evaluator, test_scheduler_integration, test_canvas_coverage, test_live_quiz_service) fail on their own merits.


- **Teacher endpoints read student-owned RLS tables via owner-guards — verify under the `meli_app` prod role (HIGH-priority to VERIFY, found in P7 planning).** Every teacher aggregate/roster/results endpoint that reads a student-owned RLS table (`checkpoint_responses`, `attendance_records`, `work_item_progress`, `activity_responses`, `readiness_responses`) — e.g. `checkpoints.py::get_checkpoint_results`, `attendance.py` roster, `checklist.py`, `scores.py::build_score_records`, `activities.py::get_activity_results` — relies on the request DB connection being able to SEE other users' rows. This PASSES in tests only because the test/app connection runs as a privileged/superuser role (BYPASSRLS per migration `28236be3d7b3`). If the production `meli_app` role does NOT have BYPASSRLS, these teacher reads will silently return EMPTY under RLS (the owner-isolation policy filters to `current_setting('app.current_user_id')`, which is the teacher's id, not the students'). **Action for final `/security-review`:** confirm the prod `meli_app` role's RLS posture; if it enforces RLS (no BYPASSRLS), either (a) grant the app role BYPASSRLS (matching test), or (b) add explicit teacher-read RLS policies, or (c) set the GUC appropriately per-read. This affects ALL phases' teacher-facing reads of student-owned tables — verify before any non-superuser prod deploy.
- **Pooled-connection RLS GUC — LESS severe than originally tracked (updated in P7 planning).** `backend/app/database.py` ALREADY has a connection-checkout reset listener (`_reset_rls_context`, ~lines 26–35) that clears `app.current_user_id` on checkout, so a pooled connection does not carry a prior request's GUC into a new checkout. P7 hardening (plan task B10) adds a symmetric check-in reset + regression test as defense-in-depth. Do NOT switch to a transaction-scoped GUC — `deps.py::get_current_user` sets it once for a multi-commit request flow, which a `is_local=true` GUC would break.
- **P6 combined-review finding (LOW, tracked):** insights cohort reads (`insights.py::_cohort_mastery_summary`, `cohort_ilo_map`, `course_effectiveness`) load all per-course mastery/outcome rows into memory with no LIMIT — fine at pilot scale, add pagination/aggregation-in-SQL for larger cohorts. (The P6 review's MEDIUM — instructor could target a follow-up at a non-enrolled `user_id` — and LOW-1 — follow-up detail missed `verify_enrollment` — were FIXED in `8f26dfa`, not deferred. LOW-2 = the teacher-cross-user-read-under-`meli_app` item already tracked above.)
- **P5 combined-review LOW findings (tracked, not blocking):** (1) `POST /activities/{id}/responses` depends on `get_current_user` (not `require_student`), so an instructor actively enrolled in their own course could submit a participation response, inflating `submission_count` in their own results view — no grade impact (score export filters `role=='student'`); matches the checkpoint-response precedent. (2) `scores.py::get_my_scores` return annotation is `APIResponse[StudentScoreRecord | None]` while the decorator `response_model` is `APIResponse[StudentScoreRecord]` — cosmetic; works because `APIResponse.data` is nullable.
- **P4 combined-review LOW findings (tracked, not blocking):** (1) `work_items.visible_from` is never enforced as a visibility gate — a future-`visible_from` teacher-authored item still shows immediately on the student checklist (`checklist.py::_build_checklist`) and calendar (`meetings.py::calendar_feed` work_item query filters only on `due_at`/`close_at`). If `visible_from` is meant as a release gate (spec §4.6), add the filter. (2) `_derive_progress_status` (`checkpoint_responses.py`) can mark a checkpoint `completed` if a card is soft-deleted AFTER a student answered it (on_time_count counts all responses; live_card_count counts only non-deleted) — correctness edge, not security. (3) Materials instructor authorization (`documents.py::_is_course_instructor`) uses an `instructor`-role enrollment rather than `Course.instructor_id` (admits co-instructors, doesn't check enrollment status) — diverges from the `get_owned_course` owner-guard used elsewhere; appears intentional (course-staff model) but flag for consistency. NOTE: the HIGH found in the same review (`meetings.py::_accessible_course` missing active-enrollment) was FIXED in `d417285` (not deferred).
- **Scan-time checkpoint status/window re-check** (found in P3 T9/T10 combined review, MEDIUM). The QR scan endpoint (`app/api/attendance.py` / `services/checkpoint_attendance.py`) gates on the *launch row* being `status='active'` + token `exp` future, but does not re-evaluate the checkpoint itself beyond `deleted_at`. If a teacher moves a checkpoint back to `draft`/`archived` while an `active` launch row lingers, a student could still scan and mark attendance for the now-non-live checkpoint. Impact LOW (participation-only; rotate/close is the normal path). Recommended fix: close active launches on any checkpoint transition away from `published`/`live`, OR add a `_assert_launchable`-style status re-check (allowing `late`) at scan time. No CRITICAL/HIGH found in the QR surface otherwise (token signing pins HS256 + enforces `exp` + ≥32-byte secret on encode AND decode; `user_id` comes from the authenticated principal, never the token/body; the three rate-limit classes are disjoint; attendance never emits mastery/learning_event).
- **Pooled-connection RLS GUC persistence** (found in P2 Task 8, affects ALL RLS tables since P0). `deps.py::get_current_user` sets `app.current_user_id` with `is_local=false` (session-level), so it persists on the pooled connection across requests. Risk is LOW/fail-closed: every authenticated request calls `get_current_user` which overwrites the GUC before any RLS-table access, and a blank/reset GUC makes `current_setting(...,true)::uuid` raise (fails closed, doesn't leak). Residual risk: a code path borrowing a pooled connection and touching an RLS table WITHOUT going through `get_current_user`. Recommended fix (defense-in-depth): `is_local=true` (transaction-scoped GUC) or explicit `RESET app.current_user_id` on connection check-in. Verify no flow depends on the GUC surviving across transactions on one connection before changing.
