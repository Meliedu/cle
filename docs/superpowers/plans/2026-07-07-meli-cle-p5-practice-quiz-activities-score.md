## Decisions locked (reconciliation with spec §4.4–4.5 + real P3/P4 code)

1. **Practice-vs-graded is a NEW `quizzes.assessment_purpose` column (`practice | graded`), NOT the existing `purpose`.** The spec §4.5 line "`quizzes.purpose` (exists) distinguishes practice from quiz" is WRONG against the real schema: `quizzes.purpose` already exists with CHECK `IN ('after_class','live')` (migration `f1c8a3b5d2e4`, index `ix_quizzes_course_id_purpose`) — that column means *when the quiz runs relative to class*, a different axis. `quiz_type` (free `String(20)`, default `practice`) is unconstrained legacy metadata. To avoid silently overloading a live column, P5 adds a **dedicated** `assessment_purpose String(20)` (CHECK `practice|graded`, `server_default='practice'` so every existing quiz backfills to practice) alongside `score_bearing bool` (`server_default=false`). The FE reads `assessment_purpose` to branch practice vs graded flows. (If a reviewer prefers reusing `quiz_type`, note it — but `quiz_type` has no CHECK and mixed historical values, so a new constrained column is safer.)
2. **Matching / ordering / short-answer are NEW question RENDERERS on the EXISTING free-string `Question.type` — NO enum widening.** `Question.type` is `String(30)` with no DB CHECK, so P5 just introduces the string values `matching`, `ordering`, `short_answer` (alongside the existing `multiple_choice`). Storage adapts the existing columns (Decision 2 requires no migration for question types): `options JSON` holds the renderable payload (matching pairs `{left:[...], right:[...]}`, ordering items `[...]`, MC option map), and `correct_answer String` (NOT NULL) holds the **JSON-encoded** answer key (a matching map, an ordered id list, or the accepted short-answer string). A backend `grade_question(question, answer)` helper (B7) decodes per type and returns a 0/1 outcome; short-answer auto-grade is exact/normalized-match only (no LLM) — richer grading is manual/out-of-scope. A tiny shape-validation guard rejects malformed `options`/`correct_answer` at question create/update.
3. **`activities` is course-scoped + teacher-authored (NO RLS, endpoint-guarded, mirrors `WorkItem`/`Checkpoint`); `activity_responses` is the student-owned RLS table (owner = `user_id`).** `activity_responses` COPIES the `work_item_progress` RLS migration `e6c2b8f4a19d` verbatim (create table + `ix_activity_responses_user_id` + `ENABLE ROW LEVEL SECURITY` + `activity_responses_owner_isolation` policy `USING/WITH CHECK (user_id = current_setting('app.current_user_id', true)::uuid)`) — the same shape as `checkpoint_responses`/`readiness_responses`/`work_item_progress`. `activities.format` CHECK = `swipe | vote | comment_reaction` (§4.4); `activities.status` CHECK = `draft | published | live | closed | archived` (mirrors the checkpoint machine); `activity_responses` unique `(activity_id, user_id)` (comment_reaction stacks multiple reactions INSIDE the `payload JSONB`, per §4.4). Publish-settings columns (§4.5) ride on `activities` too: `score_category_id FK nullable`, `points`, `grading_mode enum(auto|manual|participation)`, `open_at/due_at/close_at`, `late_rule enum(accept_late|reject_late|accept_with_flag)`, `anonymous bool`.
4. **The publish→work_item transactional seam MIRRORS P4 B4 verbatim, with `source_kind ∈ {practice, quiz, activity}`.** In BOTH the quiz publish (B5) and the activity publish (B8): resolve/gate, `_apply_transition`-or-toggle, then call `upsert_work_item(db, course_id=…, source_kind=<practice|quiz|activity>, source_id=<quiz/activity id>, title=…, required=<graded?>, score_bearing=<graded?>, due_at=due_at, close_at=close_at, created_by=user.id)` BEFORE `await db.commit()` — idempotent on the `(course_id, source_kind, source_id)` unique index, and kept-in-sync on re-publish (title/due_at/close_at) exactly like `publish_checkpoint`. `source_kind` is `quiz` when `assessment_purpose='graded'`, `practice` when `'practice'`; activities are always `activity`. `required`/`score_bearing` = `True` only for graded/score-bearing artifacts (a practice quiz is `required=False, score_bearing=False`, so `mark_missed_work_items` never marks it missed — Decision 8).
5. **The activity/quiz evidence seam MIRRORS `checkpoint_responses.py`; which kinds emit mastery vs participation-only is fixed here.** Every student submission path writes `work_item_progress` on the ANSWER'S OWN commit (via `upsert_progress`), then a best-effort block emits ONE `record_attempt_event` and (conditionally) `update_concept_mastery` — never a parallel evidence path (Global rule). Mastery emission rule:
   - **Practice + graded quiz auto-gradable questions** (`multiple_choice`, `matching`, `ordering`, and exact-match `short_answer`): emit `update_concept_mastery` per answered concept-tagged question with `outcome = grade_question(...)` (0/1), reusing the `_enqueue_mastery_for_quiz` shape (`target_kind="question"`, `attempt_kind="quiz"`). This is UNCHANGED from today's `submit_attempt` for MC — P5 just routes new types through `grade_question`.
   - **Activities (`swipe`, `vote`, `comment_reaction`) are PARTICIPATION-ONLY** — like attendance (P3), they carry no correctness signal, so they emit a `LearningEvent` (`stage="during_class"`, `source_kind="activity"`, `event_type="attempt"`) for the review/insights seam but **NEVER enqueue `update_concept_mastery`**. (An activity `config` MAY concept-tag for coverage analytics later, but P5 does not derive mastery from an opinion swipe/vote.)
6. **The live-activity monitor is a SECOND `ConnectionManager` instance in a NEW `app/services/activity_monitor.py`, copying `checkpoint_monitor.py` — no new WS framework (Decision from P3 T12).** `activity_monitor.monitor_manager = ConnectionManager()` keyed by `str(activity_id)`; `compute_activity_monitor_state(db, activity_id)` returns `{submission_count, distribution}` (a format-specific aggregate: swipe left/right counts, vote tallies, reaction histogram); `broadcast_state/submission/closed` mirror the checkpoint helpers. The WS endpoint `WS /api/activities/{id}/monitor` copies `checkpoints.py::websocket_monitor` (token → `verify_jwt` → resolve `User` → OWNER-guard the activity's course → `monitor_manager.connect` → push `state`, drain inbound). The activity response service's `_notify_monitor` seam calls `broadcast_submission` best-effort (a broadcast failure never rolls back the committed response).
7. **The score gate is a server-side `SCORE_POLICY_INCOMPLETE` typed code (Global rule §3.4), raised in the shared publish path for GRADED quiz + score-bearing activity only.** A shared `services/score_policy.py::assert_score_policy_complete(artifact)` raises `HTTPException(422, {"code": "SCORE_POLICY_INCOMPLETE", "message": …, "missing": [...]})` when a graded/score-bearing artifact is missing any required field (`score_category_id`, `points`, `grading_mode`, and a deadline `due_at`/`close_at`). Practice quizzes and participation-only activities SKIP the gate (they publish freely). The FE maps the code to a designed `StateBanner tone="blocked"` listing `missing`. **Grade export is audited:** every `GET /courses/{id}/grade-export.csv` appends one `grade_exports` row (`exported_by`, `format`, `filters JSONB`, `row_count`, `created_at` — append-only, no soft delete) BEFORE streaming, inside the same request; the export is owner-guarded (`get_owned_course`) and, per Global security rules, the endpoint is rate-limit-eligible (note it for the final gate — do not add a new rate-limit class unless abuse-prone).
8. **Work items for practice are NON-required so the missed-cron never fires on them.** `mark_missed_work_items` (P4 B9) flips only `required` past-due items to `missed`. A practice quiz publishes with `required=False, score_bearing=False` → it appears on the checklist/calendar as optional and is never auto-missed. A graded quiz / score-bearing activity publishes `required=True` → the EXISTING cron covers it with zero new code (confirm no new cron is needed — Decision confirmed against `worker.py::_run_cron_ticks`).
9. **Fold-in = MOUNT existing surfaces under Activities, do NOT rebuild.** Flashcards (`components/flashcard/`), pronunciation (`components/pronunciation/`), revision (`components/revision/`), and live-quiz (`components/live-quiz/`) already have working players/lists/hooks. The teacher Activities home and the student Activities list link/route into those existing components (new route shells + i18n only). No behavioral rebuild; P5 only adds the Activities-area entry points + the new `activities`/practice/graded-quiz surfaces.

- **`publish_quiz` currently just toggles `is_published`** → P5 rewrites it into a gated, work-item-writing publish (B5), preserving the practice toggle path.
