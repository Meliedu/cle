## Decisions locked (reconciliation with spec §4.9–4.10 + the real evidence engine + tracked findings)

1. **Reports draft ONLY from REVIEWED learning notes — no unreviewed content ever leaks (Core §0.2).** `draft_report` (B3) selects `LearningNote` rows where `review_status IN ('reviewed','edited')` **AND** `report_eligibility = true` for the course + period window. `draft`/`queued`/`merged`/`split`/`archived` notes are invisible to drafting. The report `body` sections (summary / completed work / weak points / next actions / claim limits) are built from those reviewed notes' instructor-facing fields (`observed_signal`, reviewed `draft_interpretation`, `limitation_note`) + the spine (`work_item_progress` completed counts) + `concept_mastery` weak concepts; `evidence_refs = [reviewed_note.id, …]`. **A report is created with `status='draft'` and a NON-EMPTY `evidence_refs` — if zero reviewed eligible notes exist, NO report row is created** (spec §4.9: "No report leaves `draft` without evidence refs"). The claim-limits section is `pilot.claim_limits['report']` verbatim.

2. **`reports` is a NEW table (spec §4.9); read model = student-owned RLS on the student-audience rows, teacher access via endpoint guard.** ONE `reports` table: `course_id`, `audience` CHECK `student|teacher`, `user_id` nullable (per-student weekly/end-term; NULL = teacher course-level), `period` CHECK `weekly|end_term`, `period_start`/`period_end` (tz), `body JSONB`, `evidence_refs` `UUID[]`, `status` CHECK `draft|reviewed|sent|archived`, `reviewed_by`/`reviewed_at`, `sent_at`, `export_history JSONB`; UUID PK + `TimestampMixin`. **RLS ENABLED with `reports_owner_isolation` USING/WITH CHECK `user_id = current_setting('app.current_user_id', true)::uuid`** (COPY `d94257fc717c`) so a student reads ONLY their own rows; the student endpoint (B7) further filters `audience='student' AND status='sent'`. Teacher-course-level rows have `user_id = NULL` (`NULL = GUC` is never true → students can never see them). **Teacher read/write of student-audience rows goes through `get_owned_course` endpoints, mirroring EXACTLY how `checkpoints.py::get_checkpoint_results` reads student-owned `checkpoint_responses` today** (see Reality Checks: this relies on the same production DB-role behaviour that already governs every P3/P4 student-owned table — P7 introduces NO new RLS-read pattern, and B7's RLS test asserts the student-isolation half that IS enforceable).

3. **The send gate is a server-side typed code `REPORT_NOT_REVIEWED`; send + export are always audited (Global gates rule + spec §8).** `POST /reports/{id}/send` and `/export` refuse with HTTP 409 `{error_code: "REPORT_NOT_REVIEWED"}` unless `status='reviewed'` AND `evidence_refs` non-empty (the spec §3.4 "Report can send" gate: evidence refs present + review state `reviewed` + claim-limit sections). `/approve` moves `draft→reviewed` (records `reviewed_by`/`reviewed_at`). `/send` moves `reviewed→sent` (sets `sent_at`) and the student's delivery state (S069) flips. Every approve/send/export appends an append-only `audit_events` row (B2) — NEVER `review_actions` (its `learning_note_id` is NOT NULL and a report is not note-scoped).

4. **`audit_events` is a NEW general append-only audit table (spec §8: "new append-only `audit_events` (no update/delete path)").** Mirrors the P5 `grade_exports` shape: `id`, `course_id FK`, `actor_id FK users` (no ondelete — audit FK pattern), `event_type` (e.g. `report.approve|report.send|report.export|memory.decide|checkpoint.publish`), `target_kind`, `target_id UUID`, `metadata JSONB`, `created_at`. NO `updated_at`, NO soft-delete, NO update/delete endpoint. Report send/export + memory decisions route through it; the H4 coverage check asserts every audited action writes a row. (`grade_exports` from P5 stays as-is; `audit_events` is the general log the coverage check enumerates.)

5. **Course-memory decision enum `keep|revise|reject|carry_forward` needs a NEW column — `course_record_items` has NONE today.** B8 adds a small migration: `course_record_items.decision` (String(20) CHECK `keep|revise|reject|carry_forward`, nullable = undecided), `decided_by FK users` (nullable), `decided_at` (nullable tz). `POST /memory/{id}/decide` sets these + syncs `carry_forward` bool (`true` iff decision `carry_forward`) + appends an `audit_events` row (`event_type='memory.decide'`) — spec §4.10 "recorded with `review_actions` audit"; because a CourseRecordItem may have `learning_note_id = NULL`, we use `audit_events` (Decision 4), and additionally append a `review_action` ONLY when `learning_note_id` is present (its `action_type` CHECK already includes `carry_forward`/`archive`). `reject` sets `decision='reject'` (no hard delete — the table has no `deleted_at`; it stays as an audited tombstone excluded from next-term import).

6. **Carry-forward feeds next-term setup (T023 unstub) via the checkpoint-generation context — never cross-course profiling (spec §5.6).** `GET /courses/{id}/memory/next-term-suggestions` lists `course_record_items` with `decision='carry_forward'` for the SAME course-code lineage (matched on `courses.code` / prior-term course of the same instructor — NOT by student identity). `POST /courses/{id}/setup/import-memory` copies accepted items' `relationship_summary`/`action_summary`/`outcome_summary`/`instructor_comment` into the new course's checkpoint-generation grounding context (B9 threads them into `checkpoint_generation.py::_build_context` as an extra grounding block). Memory is course-bound: no student `user_id` is carried across terms — only the reviewed instructor-authored summaries. Undecided/`reject`/`keep`-only items are NOT importable (the `MEMORY_UNDECIDED` gate refuses import of an undecided item).

7. **Pooled-connection RLS GUC finding (P2 Task 8) — the fix is defense-in-depth on top of the EXISTING checkout reset (B10).** `deps.py::get_current_user` sets the GUC session-scoped (`set_config(..., false)`), and `database.py` ALREADY resets it to `''` on every pool checkout (`_reset_rls_context`, lines 26–35), so a borrowed connection starts blank and `current_setting(...,true)::uuid` on a blank GUC raises → **fails closed** (no leak). B10's fix: (a) add a SYMMETRIC `checkin` event listener that resets the GUC when a connection returns to the pool (belt-and-suspenders so a value never lingers in the pool between requests); (b) add a regression test proving a raw pooled connection that NEVER called `get_current_user` sees a blank GUC and an RLS-table SELECT returns 0 rows / raises. We DELIBERATELY DO NOT switch to transaction-scoped `set_config(..., true)`: `get_current_user` commits multiple times (lines ~91/140/163) after setting the GUC, and a later endpoint commit would drop a transaction-local GUC before RLS-table reads — the finding's own caveat ("verify no flow depends on the GUC surviving across transactions on one connection"). Session-scoped + reset-on-both-ends is the correct minimal fix.

8. **Scan-time checkpoint status re-check (P3 T9/T10 finding) — close active launches on transition + re-check at scan (B11).** Two coordinated fixes: (a) in `checkpoint_attendance.py` scan path (after resolving the `active` launch, before writing attendance) re-evaluate the checkpoint's own status — allow `published`/`live` (and `late` window per existing rules), refuse if the checkpoint is `draft`/`archived`/soft-deleted with a typed refusal (participation is only recorded for a launchable checkpoint); (b) in `checkpoints.py`, when a checkpoint transitions AWAY from `published`/`live` (e.g. back to `draft`/`archived`), close any lingering `active` `checkpoint_launches` rows (mirror the rotate/close path) so a stale launch token can't be scanned. Idempotent; participation-only (no mastery/learning_event — unchanged).

9. **P4/P5 LOW findings are addressed in B11's roundup, not left open (P7 is the hardening gate).** P4 LOWs: (1) `work_items.visible_from` never enforced as a release gate — add the `visible_from <= now()` filter to `checklist.py::_build_checklist` AND the `meetings.py::calendar_feed` work_item query (spec §4.6 release semantics); (2) `_derive_progress_status` `completed`-on-soft-deleted-card edge — count live (non-deleted) cards consistently on both sides; (3) `documents.py::_is_course_instructor` co-instructor-vs-owner consistency — leave as intentional course-staff model but add an explanatory comment + a test pinning the intended behaviour. P5 LOWs: reconcile whatever the P5 combined-review actually logged in the roadmap "Security findings" section when P5 lands (flagged there: audited CSV grade export, `activity_responses` RLS, evidence-seam-rides-answer-commit) — re-verify each is closed or track it into the final `/security-review`.
